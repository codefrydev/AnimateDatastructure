Masterclass: Deep Dive into Dynamic Programming

1. How to spot a DP Problem

When looking at a completely new problem like "Maximum Walls", how do we know it requires Dynamic Programming? We look for the Three Golden Signatures:

🏆

Signature 1: Optimization Request

The problem asks for the "Maximum number of unique walls". When a problem asks for a Max, Min, or "number of ways", it is almost always either a Greedy Algorithm or DP.

📏

Signature 2: Sequential Ordering

The entities (Robots and Walls) exist on a 1D straight line. We can sort them from Left to Right. DP thrives when we can process decisions sequentially (Robot 0, Robot 1... Robot N).

🧱

Signature 3: The Markov Property (The Clincher!)

The rule states: "A bullet stops if it hits another robot." This is the most important clue! It means Robot 1's decision only overlaps with Robot 0 and Robot 2. Because the "future" only depends on the "immediate past", we can use DP.

2. Optimal Substructure in Action

DP is fast because of State Compression. Let's look at a real scenario: Example 2.

Robots: [2, 10]
Distances: [1, 5]
Walls: [2, 5, 7]

When deciding the best move for Robot 1 (at pos 10), DP doesn't re-evaluate the whole board.

Step 1: Robot 0 (Pos 2)

dp[0][Left] = 1 wall (at pos 2)
dp[0][Right] = 1 wall (at pos 2)

Step 2: Robot 1 (Pos 10)

"I don't recalculate the walls at pos 2! I just look at Robot 0's locked-in scores of 1, add the walls I hit (5 and 7), and determine my final max score is 3."

By summarizing the entire past into a single number (the max score) and a single condition (the direction of the previous robot), we compress the timeline into just two states per robot. This is Optimal Substructure!

3. Designing the DP State Memory

In basic DP problems, we use a 1D array: dp[i] = max score up to i.
Why isn't that enough here?

If Robot i shoots LEFT, it shoots towards Robot i-1. To calculate the walls destroyed accurately, we MUST know which way i-1 shot!

  • A If i-1 shot LEFT : The blast zones move away from each other. Simple addition!
  • B If i-1 shot RIGHT : The blast zones collide! We have to calculate a complex overlapping union.

Because the math changes completely based on the previous robot's direction, we must maintain two parallel timelines. This dictates a 2D Array:

dp[i][0]
Max score if Robot i shoots Left
dp[i][1]
Max score if Robot i shoots Right

1. The Battlefield Mechanics

Imagine an infinite 1D straight line. We are commanding Turrets (Robots) to destroy passive targets (Walls). To master this, you must respect the physics of our universe:

Limited Range

A robot's bullet travels exactly distance[i] meters.

Total Penetration

A bullet destroys every wall it touches simultaneously. Walls don't slow it down.

The Blocker Rule

A bullet stops instantly upon hitting another robot. Robots act as indestructible barriers.

2. The "Aha!" Observation

Let's look closer at the Blocker Rule. Because robots stop bullets, the infinite line isn't continuous. It is fractured into strictly isolated Zones.

The Independence Principle: What Robot i decides to do only affects the immediate zone to its Left and Right. It has absolutely zero impact on Robot i+2. This localized effect confirms we can use Dynamic Programming.

3. Phase 1: The Spatial Sorting Mandate

Dynamic Programming relies on ordered sub-problems. Since our "Zones" are spatial, we must sort our arrays by position before anything else.

// Raw Input (Chaotic)
Robots: [10, 2]
Distances: [5, 1]
Zip & Sort
// Processed Input (Ordered Left-to-Right)
Robots: [2, 10]
Distances: [1, 5]

Crucial: We must also sort the walls array! Without strict spatial ordering, concepts like "Robot i-1" and "Robot i+1" have absolutely no mathematical meaning.

5. Phase 3: Safe Transitions

When we build our array dp[i][dir], some calculations are completely independent and safe.

Safe Transition A: Robot i shoots RIGHT

Shooting right aims away from the previous robots into unknown territory. It will never interfere with decisions made by i-1.

dp[i][1] = max(dp[i-1][0], dp[i-1][1]) + walls_right[i]

Safe Transition B: Both shoot LEFT

If i-1 shot left, and i shoots left, they fire in parallel. They cannot possibly collide!

dp[i][0] = dp[i-1][0] + walls_left[i]

6. Phase 4: The Collision Math

The hardest part of the problem: What if i-1 shot RIGHT and i shoots LEFT? They might overlap!

// 1. Calculate individual reaches bounded by each other

reach_R = min(pos[i], pos[i-1] + dist[i-1])
reach_L = max(pos[i-1], pos[i] - dist[i])

// 2. Check for Overlap vs Gap

if (reach_L > reach_R) { // Disjoint (A Gap exists!)
dp[i][0] = max(dp[i][0], dp[i-1][1] + walls_left[i])
} else { // Overlap occurs!
Professor's Trick: The state dp[i-1][1] ALREADY includes walls destroyed up to reach_R.
To avoid double counting the overlapping blast zone, we ONLY add the uncounted walls from (reach_R to pos[i]].
dp[i][0] = max(dp[i][0], dp[i-1][1] + walls_in(reach_R, pos[i]))
}

7. Phase 5: Complexity & Verdict

Time Complexity

  • Sorting: O(N log N + W log W)
  • Binary Search: O(N log W)
  • DP Pass: O(N)
  • Total Time: O(N log N + W log W)

Space Complexity

  • Sorting Arrays: O(N + W)
  • DP Table: O(N)
  • Total Space: O(N + W)
